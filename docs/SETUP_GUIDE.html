03
Write the Dockerfile
Multi-stage build for a small, secure image
⏱ 10 min

The Dockerfile has two stages. This is a critical concept to understand for interviews:

Stage 1: Builder
Install ALL dependencies. Has pip, build tools, etc. This stage is thrown away after build.
Stage 2: Runtime
Copies only installed packages from Builder. No build tools. Runs as non-root user. Small & secure.
terminal — build and test the image 
# Build the image locally
$ docker build -t devops-demo-api:local .

# Check the image size (should be ~150-200MB)
$ docker images devops-demo-api
REPOSITORY         TAG     IMAGE ID       SIZE
devops-demo-api    local   abc123def456   165MB

# Run it and test health
$ docker run -p 5000:5000 --rm devops-demo-api:local
# In another terminal:
$ curl http://localhost:5000/health
{"status": "healthy", "uptime_seconds": 1.2, ...}

# Check who the container runs as (should NOT be root)
$ docker run --rm devops-demo-api:local whoami

Docker Compose + Nginx
Simulate the production architecture locally
⏱ 10 min

Docker Compose runs multiple containers together. Our local stack mirrors production:

Your Browser │ ▼ :80 ┌─────────────────┐ │ nginx │ ← Rate limiting, security headers │ (container) │ └────────┬────────┘ │ :5000 (internal network) ▼ ┌─────────────────┐ │ Flask API │ ← Gunicorn, 2 workers │ (container) │ └─────────────────┘
✅
The depends_on: condition: service_healthy in docker-compose.yml means Nginx won't start until the API passes its health check. This is production-grade container orchestration at the local level.

08
Add GitHub Secrets
Credentials for Docker Hub and server — never in code
⏱ 10 min

GitHub → Your repo → Settings → Secrets and variables → Actions → New repository secret

Secret NameHow to Get It
DOCKERHUB_USERNAME Your Docker Hub username (not email)
DOCKERHUB_TOKEN hub.docker.com → Account Settings → Security → New Access Token → name it "github-actions"
DEPLOY_HOST Your server's public IP (get a free one — see Step 10)
DEPLOY_USER Usually ubuntu (AWS) or root (DigitalOcean)
DEPLOY_SSH_KEY Paste contents of your private key: cat ~/.ssh/id_rsa
DEPLOY_PORT 22 (standard SSH port)
🔑
Generate an SSH key pair if you don't have one:
ssh-keygen -t ed25519 -C "github-actions-deploy"
Add the public key (~/.ssh/id_ed25519.pub) to your server's ~/.ssh/authorized_keys
Add the private key (~/.ssh/id_ed25519) to GitHub as DEPLOY_SSH_KEY

Interview Preparation — Own This Project
How to narrate it, what to highlight, what questions to expect

Your 90-second project pitch:

"I built a production CI/CD pipeline that fully automates the deployment lifecycle for a Flask REST API. The problem I was solving: manual deployments are error-prone, slow, and require one person with server access — a single point of failure. My pipeline runs automatically on every Git push: it executes 25+ unit tests with coverage enforcement, lints for syntax errors, then builds a multi-stage Docker image and scans it for security vulnerabilities using Trivy. If all gates pass, it SSHs into the server, pulls the new image, and restarts the container — the whole process takes under 3 minutes with no manual intervention. I configured Nginx as a reverse proxy with rate limiting and security headers, and the container runs as a non-root user. The thing I'm most proud of is the health check loop in the deploy script: it won't report success until the new container actually responds to /health — so if the app crashes on startup, the pipeline fails loudly instead of silently deploying a broken version."

Questions they'll ask — your answers:

QuestionKey Answer Points
Why multi-stage Docker build? Build tools (pip, compilers) never make it into the production image. Smaller attack surface, smaller image size (165MB vs 800MB+). Stage 1 builds, stage 2 runs.
Why Nginx in front of Flask? Gunicorn isn't meant for direct internet exposure. Nginx handles rate limiting, security headers, connection pooling, SSL termination, and protects against slowloris attacks.
How do you handle a failed deployment? The health check loop catches it — if /health doesn't return 200 within 30s, the pipeline job fails. The previous container was stopped, so you'd roll back with: docker pull image:sha-previous && docker compose up.
How do secrets stay secure? All credentials are GitHub Secrets — encrypted at rest, injected as env vars at runtime, never appear in logs or source code. The Dockerfile uses build-args, not hardcoded values.
How would you scale this beyond one server? Replace the SSH deploy step with kubectl set image (Kubernetes) or an ECS task definition update. The CI portion (test + build) is identical at any scale.
